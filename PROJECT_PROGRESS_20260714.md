# handsfree-chatting 项目进度快照

保存日期：2026-07-14

## 当前定位

`handsfree-chatting` 是一个面向 Microsoft Edge 的本地浏览器扩展，用于把用户预先准备好的多个 prompt 放入队列，并在同一个 ChatGPT 对话中依次自动发送。项目当前目标保持克制：优先解决“用户不想守在电脑旁等待上一轮回答结束，再手动发送下一条 prompt”的问题，不追求复杂 Agent、条件分支或自动读取回答内容。

当前设计原则是：**可靠性优先于智能性；宁可暂停，也不要盲目继续；宁可晚一点判断回答完成，也不要提前误判；发生异常时应尽量保留现场并停止后续发送。**

## 当前已完成版本

当前版本为 **V0.1.1**。

V0.1.0 已经实现最小可运行闭环：用户可以在插件弹窗中粘贴多个 prompt，并使用 `---PROMPT---` 作为分隔符；插件会在当前 ChatGPT 对话页面中发送第一条 prompt，等待回答完成，再继续发送下一条。插件支持开始、暂停、继续、重置，并把队列状态保存在 `chrome.storage.local` 中。

V0.1.1 修复了首次安装或重新加载插件后，旧的 ChatGPT 页面中尚未注入 `content.js` 时出现的通信错误：

`Could not establish connection. Receiving end does not exist.`

现在 `popup.js` 会先尝试通过 `chrome.tabs.sendMessage(...)` 与页面通信；若检测到接收端不存在，则使用 `chrome.scripting.executeScript(...)` 自动注入 `content.js`，再重试一次。为此，`manifest.json` 增加了 `scripting` 权限。

## 已经实际验证通过的功能

当前版本已经在 Microsoft Edge 中完成真实端到端验证。用户已经成功安装扩展，并确认可以完成以下流程：

用户打开 ChatGPT 对话页面，在插件中粘贴多个 prompt，点击开始；插件自动把第一条 prompt 写入聊天框并发送；插件检测 ChatGPT 是否正在生成；上一条回答完成后，自动发送下一条 prompt；整个过程中不需要用户持续守在电脑旁。

这意味着项目已经完成了核心技术可行性验证。当前系统能够形成完整闭环：

`prompt queue -> send -> wait -> detect completion -> send next prompt`

## 当前代码结构

`manifest.json` 负责声明 Manifest V3 扩展、所需权限、ChatGPT 域名匹配规则以及 content script。

`popup.html` 与 `popup.js` 提供用户控制面板。用户可以输入 prompt 队列、开始、暂停、继续和重置。`popup.js` 负责解析 `---PROMPT---` 分隔符、保存状态、找到当前 ChatGPT 标签页，并向 content script 发送控制消息。

`content.js` 是当前主要执行器。它负责寻找 ChatGPT 输入框、写入 prompt、寻找发送按钮、点击发送、检测回答是否开始、检测回答是否完成，以及推进整个 prompt 队列。

`README.md` 提供安装和使用说明。

## 当前回答完成判定机制

当前版本通过多个启发式信号判断回答是否完成。核心条件包括：停止生成按钮不存在、输入框仍然存在、最新 assistant 内容连续一段时间保持稳定。

当前稳定窗口为 5 秒，轮询间隔为 1 秒。等待回答开始的超时为 45 秒，单条回答最大等待时间为 30 分钟。

这个方案已经可以支持普通 ChatGPT 文本回答，但对较长工具调用、网页搜索、代码执行、文件分析等复杂任务，仍有进一步强化空间。

## 当前已知薄弱点

### 极端情况下存在重复发送风险

当前代码在点击发送按钮之后，才把该 prompt 标记为 `inFlight`。因此，在“已经点击发送”和“状态成功写入 storage”之间存在一个很小的窗口。如果此时页面、浏览器或插件异常中断，恢复后理论上可能再次发送同一条 prompt。

未来应考虑给每条 prompt 建立更明确的生命周期状态，例如：

`pending -> sending -> sent -> waiting -> completed`

同时可以为每条 prompt 生成唯一 ID 或内容 fingerprint，并在恢复时比较 ChatGPT 页面中最新 user message 是否已经等于当前 prompt，从而避免重复发送。

### 回答完成检测仍然依赖启发式规则

当前主要依赖停止按钮、输入框和文本稳定窗口。普通文本生成通常足够，但复杂工具调用期间，页面可能长时间无文本变化，也可能出现 UI 状态切换，因此存在理论上的 false completion 风险。

未来应考虑引入多信号完成检测，包括停止按钮、发送按钮状态、输入框状态、最新 assistant turn、DOM mutation 时间，以及页面中的工具运行状态。

### 当前调度器仍运行在 content.js 中

目前 `content.js` 同时承担网页执行和任务调度。当页面刷新、标签页被恢复、浏览器崩溃后恢复时，内存中的 worker 状态会丢失。虽然队列数据已经存放在 `chrome.storage.local` 中，但当前没有完整实现页面刷新后的自动恢复。

真正成熟的架构应该把职责拆分为：

`background service worker = 权威调度器`

`content.js = 网页执行器`

`popup.js = 控制面板`

这一重构适合作为 V2，而不建议在当前刚完成可行性验证时立即实施。

### ChatGPT 网页 DOM 改版是长期外部风险

当前插件依赖类似 `#prompt-textarea`、`[data-testid="send-button"]`、`[data-testid="stop-button"]` 等 selector。如果 ChatGPT 网页结构发生变化，插件可能失效。

未来可以加入 Health Check，在插件打开时明确显示：ChatGPT 页面是否识别成功、输入框是否找到、发送按钮是否找到、生成状态检测是否正常。这样可以更快判断是用户操作问题还是网页结构变化。

## 下一阶段优先改进建议

当前最推荐的下一版不是复杂 Agent，而是继续提高“可以放心离开电脑”的程度。

最值得优先加入的是 **Windows / Edge 完成与失败通知**。当全部 prompt 完成时，系统应弹出完成通知；当某条 prompt 失败或超时时，也应立即通知用户。这个功能成本低，但非常贴近项目最初目的。

紧接着建议增加 **逐条 prompt 状态显示**。界面中不再只有一个大文本框和总进度，而是可以看到每条 prompt 当前处于已完成、正在运行、等待或未开始状态。

随后建议增加 **运行日志**，至少记录时间、prompt 序号、状态变化、耗时和错误类型，而不需要保存 ChatGPT 回答内容。这样用户回来后可以知道整条队列在哪里耗时、哪里失败。

再往后才是 **重复发送保护、错误分类和更强的完成检测器**。网络断开、达到使用限制、登录失效、页面结构变化、生成超时等错误应被区分处理，而不是统一视为普通超时。

## 当前不建议立即加入的功能

暂时不建议做复杂条件分支、节点式 workflow editor、自动解析自由文本回答后决定下一条路径、完整 Agent 系统或 API 编排器。

当前项目最强的价值是简单：

**用户提前准备一串有上下文依赖的 prompt，然后离开电脑。**

在这个定位下，最重要的工程指标仍然是：

**宁可暂停，也不能重复发送；宁可晚判断，也不能提前误判完成；发生异常时必须明确告诉用户卡在哪里。**

## 当前阶段结论

截至 2026-07-14，`handsfree-chatting` 已经完成 Microsoft Edge 上的最小可行产品验证，核心自动队列功能真实跑通，初始化通信错误已经修复到 V0.1.1。当前项目已经从“概念可行”进入“有实际使用价值”的阶段。

下一次继续开发时，建议直接从本快照恢复上下文，并优先讨论 V1.1：完成/失败通知、逐条状态、运行日志，以及重复发送保护。
